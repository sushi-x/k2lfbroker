ABOUT THE CUSTOM SERVICE BROKER (DYNAMIC/DISCOVERED) PROJECT TYPE

The Custom Service Broker (Dynamic/Discovered) Project template is used to create a custom SmartObject Service Broker which uses dynamic discovery to expose a back-end provider as K2 Service Objects with Properties and Methods. This approach is normally used when you want to create a service broker which exposes a particular technology as Service Objects, but you cannot predict the schema of that underlying technology.

The dynamic SQL Server Service broker which is provided with K2 is a good example of this approach: the broker contains logic to discover the tables and stored procedures in a particular SQL database and then exposes these as Service Objects. We use a dynamic approach because we cannot predict what the schema of a particular SQL database will look like, but it’s easy to discover and describe the schema of a SQL database.

To learn more about Custom Service Brokers, please refer to the following K2 University learning module:
http://help.k2.com/en/K2LM-400BLR.aspx

You can also refer to the following K2 Underground group for samples of custom Service Broker. Many of which include source code.

DEVELOPMENT TASKS

To implement this Custom Service Broker (Dynamic/Discovered) template, you need to complete the following tasks:

1.     The target .NET Framework for the project should be set to .NET 3.5 (This should be set correctly by the project template)

2.     The Project should already have the necessary references to the SourceCode.SmartObjects.Services.ServiceSDK, but you will also find the assembly on the K2 server at C:\Program Files (x86)\K2 blackpearl\ServiceBroker\SourceCode.SmartObjects.Services.ServiceSDK.dll

3.      The Project template contains three folders, each with a single file

a.     Data\DataConnector.cs
This is where you will write all your custom code. This class is an "adapter" which implements the IDataConnector interface and is used to set up the service, discover and describe the schema of the Provider as Service Objects and perform the runtime execution for a Service Object.

b.    Interfaces\IDataConnector.cs
This is an Interface which specifies the implementations necessary for a service broker. You do not need to make any changes to this file.

c.     ServiceBrokers\ServiceBroker.cs
This is the class used by K2 to interact with your Broker. You do not need to make any changes to this class.

4.     Implement the SetupConfiguration and GetConfiguration methods, which are used to define and retrieve the Service Instance configuration values for your service. Add private members for each configuration property so they are easier to obtain at runtime.

5.     Implement the SetupService method which is used to define the service instance

6.     Implement the DescribeSchema method which is used to discover the schema of the underlying provider and expose it as Service Objects. Use the sample code in the project template for guidance.

7.     Implement the SetTypeMappings method, which is used to "translate" the data types of your underlying provider to equivalent Service Object property types.

8.     Implement the Execute method, which is used to execute Service Object methods at runtime. You will need to interrogate the method parameters and then perform the runtime interaction with the underlying provider according to the method/Service Object requested. Use the sample code in the project template for guidance.

9.     Implement the Dispose method to clean up any resources, if needed.

DEPLOYING AND INSTALLING CUSTOM SERVICE BROKERS

Depending on your K2 environment, you may need to copy the resulting .dll to several machines. The custom service broker assembly (.dll) must be copied to each physical K2 Server, in the C:\Program Files (x86)\K2 blackpearl\ServiceBroker directory. Once the assembly is copied, you need to register the assembly as a Service Type with K2 and then register a Service Instance of the Broker. This should expose Service Objects, but you will need to create K2 SmartObjects before you will be able to use the Service Objects at runtime. The procedure is described in more detail below.

1)     Copy the assembly to the C:\Program Files (x86)\K2 blackpearl\ServiceBroker directory of each physical K2 server

2)     Use the SmartObject Service Tester utility ((C:\Program Files (x86)\K2 blackpearl\Bin\SmartObject Service Tester.exe) to register a new Service Type and select your custom Broker from the list.

3)     Using the SmartObject Service Tester utility, register a new Service Instance of your Service Broker, and provide the necessary security configuration and any other Broker-specific configuration values.

4)     Once the Service Objects are discovered and added, you may use the SmartObject Service tester utility to generate SmartObjects for the Service Objects, or you may manually create SmartObjects in the K2 design tools and point them to your Service Object methods.

Note:
For testing purposes, we recommend using the utility to generate SmartObjects since it saves you time, but for the "real" SmartObjects that are used by workflow and forms designers, we recommend creating SmartObjects with the K2 Design tools.

Note:
It is not necessary to unregister the Service Type if you make changes. Just restart the K2 Service, copy your updated assembly and refresh the Service Type and the Service Instances for your Service Type. Note that existing SmartObjects that use your broker's Service Objects may break if the underlying Service Objects have changed.

 

CUSTOM BUILD CONFIGURATIONS

To assist with developing the Custom Service Broker project, the following custom build configurations are available.

 

Note: the build configurations do not register or refresh the Broker or Service Instances, You will need to perform those actions yourself. We recommend using the SmartObject Service Tester utility (C:\Program Files (x86)\K2 blackpearl\Bin\SmartObject Service Tester.exe) to perform these tasks.


Note: You may need to edit the pre- and post-build commands to copy the output files to the correct location for your environment. The custom build configurations are only set up for scenarios where your development environment and K2 environment are on the same machine (for example, when you develop using one of the standard K2 virtual machines, or if you have a local development server)

 

DebugCopyNoRestart

Stops the K2 service and copies the service broker .dll file and .pdb file to the local K2 server's ServiceBroker directory. The service is NOT restarted once the assembly is copied: use this approach if you want to start K2 in console mode to test your custom service broker with console output.

 

DebugCopyLocal

Restarts the K2 service and copies the service broker .dll file and .pdb file to the local K2 server.



DEBUGGING THE CUSTOM SERVICE BROKER

To debug a custom service broker, you will need to attach to the K2Server.exe process and use remote debugging. See details below.

 

Runtime debugging (K2 Studio/K2 for Visual Studio)

a)     Add breakpoints to your code

b)    Stop the K2 service

a)     Build and then copy the assembly .dll file and .pdb file to the K2 Host Server's ServiceBroker directory ([Program Files]\K2 blackpearl\ServiceBroker)

b)    Start the K2 service

c)     Use the Visual Studio Debug > Attach to Process menu to attach to the K2HostServer.exe process

d)    Use the SmartObject Service Tester utility ([Program Files]\K2 blackpearl\Bin\ SmartObject Service Tester.exe) to perform the operation you wish to debug (for example, registering a service instance or executing a SmartObject which uses a Service Object in your broker)

e)     Your breakpoint should be hit

 

SmartObject Logging Configuration

You can also configure SmartObject logging to output logging information at runtime.


Note: we do not recommend enabling SmartObject logging in production environments since it may affect performance. You should only enable logging for troubleshooting purposes.

a)     Stop the K2 service

b)    Make a backup of the following file: [Program Files]\K2 blackpearl\Host Server\Bin\SourceCode.SmartObjects.Runtime.config

c)     Open the following file in a text editor: [Program Files]\K2 blackpearl\Host Server\Bin\SourceCode.SmartObjects.Runtime.config

d)     Change all the
log=no
items in the <Logging> section to
log=yes
(The "Overwrite" setting for each log file will overwrite the log file every time a SmartObject method is executed. You may or may not want to enable this overwrite function)

e)     Restart the K2 blackpearl Server service to pick up the configuration changes

f)     Use the SmartObject Service Tester utility ([Program Files]\K2 blackpearl\Bin\ SmartObject Service Tester.exe) to perform the operation you wish to log (for example, registering a service instance or executing a SmartObject which uses a Service Object in your broker)

g)    The logging information will be output to the following location: C:\Program Files (x86)\K2 blackpearl\ServiceBroker\logs

                i.    brokerpackagein.log logs the SmartObject that called the broker and the method of the SmartObject that uses the Broker. This information is useful to see which SmartObject and method were executed.

               ii.    brokerpackageout.log only contains a number of records that were returned by the broker. This data can be useful when you need to troubleshoot why not all the expected records were returned to the UI. For example: perhaps the UI was filtering out data after the broker returned the expected number of records.

              iii.    serviceauth.log shows the security credentials used when the service was execute. Check the "current thread user" to see which user account execute the service broker code.

              iv.    servicepackagein.log represents the Service Instance configuration, the Service Objects for the Service Instance that were called along with their Methods and Properties, the Mappings that define how the Provider Types are mapped to Service Object Types.
When troubleshooting, one of the most useful values is probably the <serviceauthentication> section, where you can see the user account context that was used to query the provider.

                i.    servicepackageout.log is the "raw" data returned by the service instance. This information can be useful when the data which eventually surfaces to the UI does not match the data you were expecting. Check this log file to see if the expected data was returned to the SmartObject, in which case the problem is likely on the UI side
